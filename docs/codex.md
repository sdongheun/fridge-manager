# Codex 작업 규칙

## 목적
- 이 문서는 이 저장소에서 Codex가 어떻게 작업해야 하는지 정의한다.
- 각 작업용 스레드는 먼저 이 문서를 읽고, 이후 현재 작업에 필요한 문서와 소스 파일만 추가로 읽는다.

## 기본 작업 순서
1. 이 문서를 읽는다.
2. `docs/mvp.md`를 읽는다.
3. 현재 작업과 관련된 영역 문서만 읽는다.
   - UI, 화면, 컴포넌트, 스타일링: `docs/frontend.md`
   - 저장소, 데이터 모델, 영속성, 향후 API 구조: `docs/backend.md`
4. 현재 작업과 직접 관련된 소스 파일만 확인한다.
5. 작업을 끝까지 완료할 수 있는 가장 작은 합리적 변경을 적용한다.

## 작업 원칙
- 한 번에 하나의 구체적인 작업만 진행한다.
- 사용자가 명시적으로 요구하지 않으면 앱 전체를 재설계하지 않는다.
- 추상적인 확장 설계보다 점진적인 구현을 우선한다.
- 명확한 이유가 없다면 기존 프로젝트 스타일을 유지한다.
- 앞으로의 작업에 영향을 주는 결정이 생기면 관련 문서를 짧게 갱신한다.
- 구현이 필요한 스레드는 객체지향적인 책임 분리를 의식해 코드를 작성한다.
- 코드는 주니어 개발자가 읽고 이해하기 쉽게 작성한다.
- `any` 타입 사용은 지양하고, 가능한 한 명시적인 타입을 사용한다.

## 응답 원칙
- 개념 질문에는 이 프로젝트 기준에서 실무적으로 설명한다.
- 구현할 때는 필요한 가정을 짧게 밝히고 바로 수정한다.
- 모호함이 있어도 안전한 기본값이 있으면 그 값을 택해 계속 진행한다.

## 기준 우선순위
1. 현재 스레드에서 사용자가 직접 준 지시
2. `docs/mvp.md`
3. `docs/frontend.md`, `docs/backend.md`에 적힌 영역별 규칙
4. 저장소 안에 이미 존재하는 코드 패턴

## 브랜치와 스레드 원칙
- 가능하면 스레드 하나에 작업 하나만 맡긴다.
- 의미 있는 구현 단위마다 브랜치를 분리한다.
- 브랜치 이름은 기본적으로 `codex/` 접두사를 사용한다.
- 관련 없는 변경을 한 브랜치에 섞지 않는다.

## 토큰 효율 원칙
- 필요하지 않으면 저장소 전체를 읽지 않는다.
- 이미 필요한 맥락이 확보된 긴 파일을 반복해서 읽지 않는다.
- 현재 질문이나 작업에 필요한 파일만 읽는다.
- 범위를 넓히기 전에 현재까지 파악한 맥락을 짧게 정리한다.

## 문서 갱신 규칙
- `docs/mvp.md`는 제품 범위나 방향이 바뀔 때만 수정한다.
- `docs/frontend.md`는 UI 구조나 프론트엔드 규칙이 바뀔 때 수정한다.
- `docs/backend.md`는 데이터 구조, 저장 방식, API 경계가 바뀔 때 수정한다.
- 문서는 짧고, 구체적이고, 실제 작업에 바로 쓸 수 있게 유지한다.
