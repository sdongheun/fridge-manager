# 백엔드 및 데이터 가이드

## 범위
- 이 프로젝트의 MVP 백엔드는 Supabase Auth + Postgres 기준으로 설계한다.
- 이 저장소에서 "백엔드"는 인증, DB 스키마, RLS, 저장 계층 경계, 향후 API로 확장 가능한 구조를 의미한다.

## 현재 가정
- 인증은 Supabase Auth 이메일/비밀번호 기반으로 시작한다.
- 데이터 저장은 Supabase Postgres를 기준으로 한다.
- 날짜 처리는 `dayjs`를 사용한다.
- 앱은 초기에 온라인 동작을 기준으로 하되, 오프라인 동기화는 MVP 범위에서 제외한다.

## 데이터 모델 방향
- 인증 주체는 `auth.users`이며 앱 전용 정보는 `profiles`에 둔다.
- 사용자당 냉장고는 하나를 기본값으로 두고 `fridges.owner_user_id`로 소유자를 구분한다.
- 핵심 엔티티는 `food_items`이며 사용자 분리를 명확히 하기 위해 `user_id`와 `fridge_id`를 함께 저장한다.
- `food_items` MVP 필드:
  - `id`
  - `user_id`
  - `fridge_id`
  - `name`
  - `expiry_date`
  - `purchase_date`
  - `quantity`
  - `unit`
  - `category`
  - `storage_zone`
  - `barcode`
  - `memo`
  - `created_at`
  - `updated_at`

## 저장 규칙
- Supabase 접근은 기능 단위 모듈 뒤로 숨긴다.
- UI 코드에서 인증 호출과 테이블 쿼리를 직접 반복하지 않는다.
- `auth`, `foods`처럼 기능 기준으로 저장/조회 함수를 나눈다.
- 사용자 ID는 클라이언트 입력이 아니라 현재 세션 기준으로만 채운다.
- 백엔드 로직도 한 파일에 몰아넣지 말고, 기능/도메인/저장 책임 기준으로 분리한다.
- 가독성을 높이기 위한 모듈 분리를 우선하고, 과도한 추상화는 피한다.

## 데이터 동작 규칙
- 유통기한 계산 로직은 유틸 함수에 모아둔다.
- 정렬 규칙은 결정적이고 설명 가능해야 한다.
- 생성 및 수정 흐름에서는 최소 필수값을 검증한다.
- 스키마 변경은 SQL 마이그레이션으로 남기고 문서도 함께 갱신한다.
- 저장 계층과 도메인 로직은 객체지향적인 책임 분리를 의식해 구성한다.
- 주니어 개발자가 이해하기 쉬운 이름과 데이터 흐름을 유지한다.
- `any` 대신 명확한 도메인 타입과 입출력 타입을 정의해 사용한다.

## 인증 흐름
- 회원가입 시 `auth.users` 생성 후 트리거로 `profiles`, 기본 `fridges` 레코드를 만든다.
- 로그인은 이메일/비밀번호 기반 세션 생성으로 시작한다.
- 로그아웃은 현재 세션만 종료한다.
- 앱에서 사용자 표시 정보가 필요하면 `profiles.display_name`을 우선 사용한다.

## RLS 규칙
- `profiles`: 본인만 조회/수정 가능
- `fridges`: 본인 소유 냉장고만 조회/수정 가능
- `food_items`: `auth.uid() = user_id` 이고 연결된 `fridge_id`도 본인 소유일 때만 조회/변경 가능
- 초기 협업 기능이 없으므로 공유 냉장고 정책은 넣지 않는다

## 바코드 저장 기준
- MVP에서는 별도 바코드 마스터 테이블을 만들지 않는다.
- 스캔 결과가 있으면 `food_items.barcode`에 문자열로 저장한다.
- 상품 메타데이터 캐시나 외부 바코드 카탈로그 연동은 실제 요구가 생길 때 추가한다.

## 알림 기준
- 알림용 기준 데이터는 별도 테이블보다 `expiry_date`를 기준으로 계산한다.
- 기본 규칙은 `expiry_date - today <= 3일`인 항목을 임박 대상으로 본다.
- 만료 항목은 `expiry_date < today`로 판단한다.
- 사용자별 알림 on/off 또는 시간대 설정이 필요해질 때만 별도 설정 테이블을 추가한다.

## 향후 백엔드 확장 대비
- 나중에 서버가 추가되더라도 도메인 타입은 가능한 한 안정적으로 유지한다.
- 저장 구현 세부사항과 도메인 로직은 너무 강하게 결합되지 않게 분리한다.
- 실제 API 계층을 넣기 전에는 repository/service helper 수준으로 유지한다.

## 백엔드에서 해야 할 것
- 데이터 경계를 명확히 유지한다.
- 저장소 코드는 테스트 가능하고 분리된 형태로 유지한다.
- 스키마와 정책은 설명 가능할 정도로 단순하게 유지한다.
- 스키마 변경이 생기면 이 문서에 짧게 기록한다.

## 백엔드에서 피할 것
- 아직 별도 서버 아키텍처를 설계하지 않는다.
- MVP에 복잡한 캐싱, 동기화, 이벤트 버스를 넣지 않는다.
- 날짜 파싱 규칙이 여러 파일로 흩어지지 않게 한다.
