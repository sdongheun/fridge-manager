# 프론트엔드 가이드

## 범위

- 이 문서는 화면, 컴포넌트, 스타일링, 클라이언트 상태 흐름, 사용자 상호작용 규칙을 다룬다.

## 현재 스택

- Expo
- React Native
- TypeScript
- Expo 설정 기반 ESLint

## 현재 구조

- 앱 진입점: `App.tsx`
- 화면 계층: `src/screens`
- 기능별 코드: `src/features`
- 공용 상수: `src/constants`

## 프론트엔드 원칙

- 화면은 단순하고 읽기 쉽게 유지한다.
- 재사용 가능한 도메인 로직은 `src/features`로 분리한다.
- 암묵적인 객체 형태보다 명시적인 TypeScript 타입을 선호한다.
- 아주 작은 UI 조각에 대해서는 성급한 추상화를 피한다.
- MVP 단계에서는 화려함보다 직접적이고 동작하는 UI를 우선한다.
- 객체지향적인 책임 분리를 의식해 코드를 작성한다.
- 주니어 개발자가 읽고 이해하기 쉬운 이름과 흐름을 우선한다.
- `any` 대신 구체적인 타입, 인터페이스, 유니온 타입을 우선 사용한다.
- 기능 전체를 한 파일에 몰아넣지 말고, 화면/컴포넌트/도메인 로직을 책임 기준으로 분리한다.
- 컴포넌트 분리는 가독성을 높이는 방향으로만 적용하고, 불필요한 쪼개기는 피한다.

## 스타일링 규칙

- 가능하면 공용 상수의 토큰을 재사용한다.
- 의도적인 리디자인 작업이 아니라면 현재 앱의 시각적 톤을 유지한다.
- 화면 단위 컴포넌트는 `StyleSheet.create` 기반 스타일을 우선한다.
- 정말 일회성 값이 아니면 인라인 스타일을 남발하지 않는다.

## 상태 관리 규칙

- MVP에서는 가장 단순하게 동작하는 상태 관리 방식을 우선한다.
- 범위가 한 화면 안에 갇혀 있다면 로컬 상태를 사용해도 된다.
- 냉장고 식재료 상태를 여러 화면이 함께 쓰기 시작할 때 공용 상태로 분리한다.
- 유통기한 라벨 같은 파생 값은 UI 코드에 중복 계산하지 말고 유틸 함수에 둔다.

## 화면 구성 기대치

- 홈 화면은 냉장고 식재료 목록의 기본 화면이 되어야 한다.
- 추가와 수정 흐름은 작업 범위에 따라 별도 화면 또는 단순 모달 흐름으로 시작할 수 있다.
- 빈 상태는 명확하고 도움이 되게 보여야 한다.
- 유통기한 정보는 상세 화면에 들어가지 않아도 바로 읽혀야 한다.

## 프론트엔드에서 해야 할 것

- UI는 MVP 범위 안에서만 구성한다.
- 컴포넌트 이름은 도메인 중심으로 짓는다.
- 날짜 계산과 유통기한 계산은 유틸 함수를 사용한다.
- 읽기 쉬운 라벨과 충분한 터치 영역 같은 기본 접근성을 유지한다.

## 프론트엔드에서 피할 것

- 아직 큰 디자인 시스템을 도입하지 않는다.
- 실제 필요가 없으면 무거운 상태 관리 라이브러리를 추가하지 않는다.
- 저장소 로직을 여러 UI 컴포넌트에 직접 흩뿌리지 않는다.
- 작업 요구가 없다면 고급 애니메이션을 추가하지 않는다.
